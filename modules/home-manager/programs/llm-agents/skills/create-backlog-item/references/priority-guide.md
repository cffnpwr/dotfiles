# Priority Guide

`Priority` フィールドに入れる値の目安。`create-backlog-item` で値を決めるときの参照用。

## 大原則

- **0--100 の整数**
- **小さいほど高優先度** (0が最優先、100が最低)
- **デフォルト値は設けない**。迷ったらデフォルト値に逃げずに、必ず分類を判断して帯を決める

`plan-sprint` はこの値の昇順でバックログを引くので、上に来てほしいものほど小さく付ける。

## 帯 (Band)

| 範囲 | 区分 | 使うのはどんなとき |
|---|---|---|
| `0--9` | 特例 | 本当に今すぐ対応しなければいけないもの。本番障害発生中、active exploit が確認されている脆弱性、データ消失リスク。**普段は使わない帯** |
| `10--19` | 即時対応セキュリティ・不具合 | プロダクション影響あり / アクティブな悪用 / データ破損 |
| `20--29` | 高優先度セキュリティ・不具合修正 | ユーザーに見える影響 / 回避策が貧弱 |
| `30--39` | 期日のあるタスク | カテゴリ不問。外部期日 (リリース日、EOL、コンプライアンス期限) があるタスクはここに上げる |
| `40--49` | 即時影響のないセキュリティ・不具合修正 | エッジケース / 軽微なUI崩れ / 内部からしか踏めない経路 |
| `50--59` | 新機能実装 | |
| `60--69` | 依存パッケージアップデート | |
| `70--79` | ドキュメント整備 | |
| `80--89` | 機能改善・リファクタリング | |
| `90--99` | リポジトリ整理等の雑務 | |
| `100` | いずれ着手すればよいこと | 仮説検証前のアイディア、永久保管枠 |

### 期日帯 (30--39) の扱い

「期日のあるタスクは30--39」は **本来の帯より高い場合に上げる** ためのルール。既に 0--29 にあるタスク (例: 即時対応セキュリティ) はそのまま — 期日があっても 30--39 に下げてはいけない。

## 帯内の値の付け方

基本は **帯の最小値** (例: `50--59` 帯なら `50`)。期日や着手の容易さで上に振る。

下端寄り (帯の最小値):

- 期日が近い
- 影響範囲が広い
- 着手が難しい (タイミング限定、まとまった時間が必要)

上端寄り (帯の最大値):

- 期日がない、または遠い
- 影響範囲が局所的
- 着手が容易 (すぐ終わる、依存が少ない)

## 派生タスクの帯判定

テスト追加・CI/CD整備・spike (調査) は専用の帯を持たない。**作業の本質** で帯を決める:

| 派生タスク | 判定 |
|---|---|
| 新機能のテスト追加 | 新機能実装 (50--59) |
| 既存機能のテスト整備 | 機能改善・リファクタリング (80--89) |
| バグ再現テスト | そのバグと同じ帯 |
| CIパイプラインの新規整備 | 新機能実装 (50--59) |
| CI高速化・整理 | 機能改善・リファクタリング (80--89) |
| CI起因のバグ修正 | バグの帯 (10--19 / 20--29 / 40--49) |
| 「Xの方針を決めるための調査」 | 紐づく X と同じ帯 |

調査だけで終わるタスクは原則として作らない (調査の先に必ず着手すべき作業があるはず)。

## Priority と実行順は別

Priority は **そのアイテム単体で見たときの優先度** であって、「Priority が高い順に必ず着手する」ルールではない。`plan-sprint` のソートには使われるが、実行順はスプリント中に以下も加味して決まる:

- ブロック関係 (低優先度でも下流をブロックしているなら先に倒す)
- まとめて着手したほうが効率的なグループ
- スプリント内のリズム / コンテキストスイッチコスト

ブロッカー解消や一括処理のために Priority を本来の帯から動かす必要はない。**Priority は素直に分類と期日に基づいて決め、実行順はスプリント中に判断する**。

## アンチパターン

- 0--9 を頻繁に使う (本当の特例にしか使わないからこそ意味がある)
- 「迷ったから50」「とりあえず30」のように分類せずに値を入れる
- ブロッカー解消や一括処理のために Priority を本来の帯から動かす
- 期日付きタスクを元のカテゴリ帯のまま放置 (期日があるなら 30--39 に上げる)
- 既に 0--29 にあるタスクを「期日があるから」と 30--39 に下げる (期日帯は床、上限ではない)
