# Solo Sprint Spec

Cross-skill reference for the `solo-sprint` family. Holds only the resources used by **two or more** skills. Skill-specific resources (templates used by exactly one skill) live in that skill's own directory.

## Layout

```
solo-sprint-spec/
├── SKILL.md                    # this file
└── references/
    ├── rules.md                # sprint rules, estimation, PBI conditions, capacity
    ├── data-model.md           # Project fields, Issue body schema, storage / config.toml
    ├── gh-commands.md          # gh CLI / GraphQL / Contents API / Refs / Pulls reference
    └── path-templates.md       # template variables and date format tokens
```

## Configuration

All skills read settings from:

```
${XDG_CONFIG_HOME:-$HOME/.config}/solo-sprint/config.toml
```

Generated by `bootstrap-solo-sprint`. Skills MUST fail with a clear error if the file does not exist, instructing the user to run `bootstrap-solo-sprint` first.

## When this skill is referenced

Other solo-sprint family skills reference these documents by relative path:

- `references/rules.md` — sprint length, estimation, PBI/SBI distinction
- `references/data-model.md` — Project field IDs and option IDs, storage configuration
- `references/gh-commands.md` — concrete `gh` and `gh api` invocations (incl. Contents/Refs/Pulls APIs)
- `references/path-templates.md` — variable substitution and date format tokens for `storage.paths.*` and `storage.branch_template`

Skill-specific templates are owned by their consuming skill:

- `create-backlog-item/templates/{pbi,sbi}.md`
- `plan-sprint/templates/calendar.md`
- `retrospect-sprint/templates/retro.md`

## Anti-patterns

- ❌ Adding skill-specific assets here (they belong in the consuming skill's directory)
- ❌ Hardcoding values that should come from `config.toml`
- ❌ Inventing field names or option values not defined in `data-model.md`
- ❌ Calling this skill directly as a slash command (it is reference-only)
