# パッケージ管理機構

パッケージはOSごとにネイティブな手段で導入する。
導入対象の意図は単一のデータに集約し、両OSへ供給する。
コマンドラインツールとGUIアプリケーションを各OSの標準的なパッケージ管理に集約し、導入経路を分散させない。

## データ構成

導入対象は`.chezmoidata`にパッケージ単位で一覧化する。
各エントリは次を持つ。

- 対象OS(どのOSで導入するか)
- 導入手段(後述の導入経路のいずれか)
- 導入手段ごとに名前が異なる場合のOS別の名前

パッケージを単位とし、対象OSを各エントリに持たせることで、両OSで導入するものも一度だけ宣言でき、二重管理を避ける。

## 導入経路

各パッケージは導入手段として次のいずれかを持つ。

1. 標準のパッケージ管理(macOSは[Homebrew](https://brew.sh/)、debian系Linuxは[apt](https://wiki.debian.org/Apt)の既定リポジトリ)
2. サードパーティのリポジトリ(Homebrewのtap、または署名鍵を登録した外部aptリポジトリ)
3. リポジトリ以外からの導入(リリース配布物の取得、またはフォントなどのファイル配置)

## macOS

macOSを対象とするエントリから`Brewfile`をテンプレート生成し、`brew bundle`で適用する。
導入手段に応じてformula、cask、Mac App Storeアプリ、tapを出し分ける。
未宣言のものはcleanupで削除し、不要になった依存は`brew autoremove`で除去する。
これにより宣言した集合だけが残る。`Brewfile`は手書きせず生成物として扱う。

## debian系Linux

debian系Linuxを対象とするエントリからaptで導入する
標準リポジトリはリリースとコンポーネントを固定し、外部リポジトリは署名鍵を登録して追加する。
導入は次の順で行う。

1. ブートストラップに必要な最小パッケージの導入
2. 外部リポジトリの登録(署名鍵を含む)
3. パッケージのインストール

リポジトリの状態を整えてからまとめてインストールすることで、導入の段階を明確にする。

## バージョン方針

パッケージのバージョンは各導入手段が提供する版に追従し、特定時点へは固定しない。これは設計原則の冪等性と両立する。
特定のバージョンへ固定する必要がある場合のみ、個別に固定する。
